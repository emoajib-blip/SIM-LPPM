# Implementation Plan: Add Kaprodi as Initial Approval Gate for Research & Community Service Proposals

## Overview
Add Kaprodi (Department Head) as an initial approval gate that validates proposals before they enter the existing workflow. After kaprodi approval, proposals proceed normally through the current system (Dekan → Kepala LPPM → Reviewers).

**Key Difference**: This is NOT inserting kaprodi into the existing status workflow, but adding a pre-gate that must be passed before proposals can be SUBMITTED to the main workflow.

## Current System Analysis
- **Existing Roles**: superadmin, admin lppm, kepala lppm, dekan, dosen, reviewer, rektor
- **Current Workflow**: DRAFT → SUBMITTED → APPROVED (Dekan) → WAITING_REVIEWER
- **Faculty Structure**: Faculty has dean_user_id, StudyProgram belongs to Faculty
- **No Kaprodi Role**: Study programs don't have assigned heads currently

## Proposed Simplified Approach

### 1. Kaprodi as Pre-Submission Validator
**Concept**: Kaprodi approval happens BEFORE proposal submission, similar to team member approvals
- Kaprodi validates proposal quality and alignment with department goals
- Only after kaprodi approval can lecturer submit proposal to the main workflow
- Kaprodi approval is tracked separately from main proposal status

**Benefits**:
- No disruption to existing SUBMITTED→APPROVED workflow
- Existing approved proposals remain valid
- Kaprodi acts as quality gate, not workflow bottleneck
- Easier to implement and maintain

### 2. Database Schema Changes
**New Table: proposal_kaprodi_approvals**
```sql
CREATE TABLE proposal_kaprodi_approvals (
    id UUID PRIMARY KEY,
    proposal_id UUID NOT NULL REFERENCES proposals(id),
    kaprodi_user_id UUID NOT NULL REFERENCES users(id),
    status ENUM('pending', 'approved', 'rejected') DEFAULT 'pending',
    notes TEXT NULL,
    approved_at TIMESTAMP NULL,
    rejected_at TIMESTAMP NULL,
    created_at TIMESTAMP,
    updated_at TIMESTAMP
);
```

**Update Study Programs Table**
- Add `kaprodi_user_id` field to assign kaprodi to study programs

### 3. Role & Permission System
**Add Kaprodi Role**
- Add 'kaprodi' role to RoleSeeder
- Add permissions: 'module_persetujuan_kaprodi', 'module_proposal_kaprodi'
- Allow users to have both 'dosen' and 'kaprodi' roles

### 4. Kaprodi Approval Logic
**Create KaprodiApproval Model & Logic**
- Model to track kaprodi approvals
- KaprodiApprovalAction similar to team member approvals
- Validation: kaprodi can only approve proposals from their study program

**Integration with Proposal Submission**
- Before proposal can transition to SUBMITTED status, must have kaprodi approval
- Add validation in proposal submission logic
- Show kaprodi approval status in proposal draft view

### 5. User Interface Updates
**Kaprodi Dashboard**
- List of proposals awaiting kaprodi approval from their study program
- Approve/Reject buttons with notes field

**Lecturer Proposal Interface**
- Show kaprodi approval status in proposal form
- Prevent submission if kaprodi approval pending/rejected
- Request kaprodi approval button

**Admin Interface**
- Assign kaprodi to study programs
- View kaprodi approval statistics

### 6. Notification System
**Kaprodi Notifications**
- Notify kaprodi when lecturer requests approval
- Notify lecturer when kaprodi approves/rejects
- Notify lecturer when kaprodi approval is required

### 7. Backward Compatibility
**Existing Proposals**
- All existing proposals are automatically considered as having kaprodi approval
- No retroactive approval needed
- Kaprodi gate only applies to new submissions

**Graceful Degradation**
- If study program has no assigned kaprodi, allow submission with admin override
- Configurable per study program/faculty

## Implementation Safety Analysis

### Safe Implementation Steps (Most Conservative Order)

1. **Phase 1: Foundation (Zero Risk)**
   - Add kaprodi role to RoleSeeder ✅
   - Add kaprodi_user_id to study_programs table ✅
   - Update StudyProgram model ✅
   - Create admin interface for kaprodi assignment ✅

2. **Phase 2: Kaprodi Approval System (Isolated)**
   - Create proposal_kaprodi_approvals table ✅
   - Create KaprodiApproval model and logic ✅
   - Build kaprodi dashboard (doesn't affect existing flow) ✅

3. **Phase 3: Integration Points (Controlled Risk)**
   - Add kaprodi approval validation to proposal submission
   - Update proposal draft UI to show kaprodi status
   - Add kaprodi approval request functionality

4. **Phase 4: Notifications & Polish (Low Risk)**
   - Implement notification system
   - Add UI indicators and status displays
   - Update documentation

### Risk Assessment

**LOW RISK COMPONENTS:**
- Adding kaprodi role (doesn't affect existing users)
- Database schema changes (new tables, nullable fields)
- Kaprodi dashboard (isolated new functionality)
- Admin assignment interface (doesn't affect proposal flow)

**MEDIUM RISK COMPONENTS:**
- Proposal submission validation (could temporarily block submissions if buggy)
- UI changes to proposal forms (cosmetic, easily revertible)

**HIGH RISK COMPONENTS:**
- None - kaprodi approval is pre-gate, not integrated into core workflow

### Rollback Strategy
- **Immediate Rollback**: Remove kaprodi validation from submission logic
- **Gradual Rollback**: Disable kaprodi requirement per study program
- **Complete Removal**: Delete kaprodi approval records (no data loss)
