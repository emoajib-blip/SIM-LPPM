# Plan: Include Draft Proposals in Lecturer Quota Enforcement

## Objective
Implement functionality where draft proposals count towards lecturer eligibility quotas (head/member limits). This prevents lecturers from creating multiple draft proposals if they exceed their quota limits, and disables the "Create Proposal" button when ineligible.

## Current State Analysis
- **Eligibility Rules**: Currently implemented in `IdentityEligibilityAction` with checks for active proposal statuses (SUBMITTED, APPROVED, etc.) but excludes DRAFT status
- **Quota Types**: 
  - Per-scheme: `max_proposals_as_head/member` 
  - Global: `max_total_proposals_as_head/member` (recently added)
- **UI**: No current enforcement of eligibility before allowing proposal creation
- **Draft Status**: Proposals with status DRAFT are considered inactive for quota purposes

## Proposed Changes

### 1. Update Eligibility Logic
- Include `ProposalStatus::DRAFT` in active statuses for quota calculations
- This ensures draft proposals count against lecturer quotas

### 2. Add Pre-Creation Eligibility Check
- Create method in `EligibilityService` to check if user can create new proposal
- Call this in `ProposalCreate::mount()` to prevent access if ineligible
- Return early with error message if quota exceeded

### 3. Customizable Feedback Messages
- Create `quota_messages` table for admin-managed feedback messages
- Fields: `key` (e.g., 'button_tooltip', 'access_denied'), `message` (with placeholders like {limit}, {current})
- Admin CRUD interface for managing messages
- Fallback to default messages if not set

### 4. Update UI Components
- Add eligibility check to proposal index/create pages
- Disable "Create Proposal" buttons when user exceeds quotas
- Show warning messages using admin-customizable text:
  - Button tooltip: "Kuota terbatas: maksimal {limit} usulan sebagai ketua aktif (termasuk draft)"
  - Error message on access attempt: "Anda telah mencapai batas maksimal {limit} usulan sebagai ketua. {custom_message}"
  - Dashboard indicator: "Kuota Ketua: {current}/{limit} (termasuk {draft_count} draft)"

### 4. Update Views
- Add conditional rendering for create buttons based on eligibility
- Display quota status in user dashboard

## Implementation Steps

### Phase 1: Backend Logic Updates
1. Update `IdentityEligibilityAction::execute()` to include DRAFT in active statuses
2. Add `EligibilityService::canCreateProposal(User $user, string $type): array` method
3. Update `ProposalCreate::mount()` to check eligibility and abort if ineligible

### Phase 2: Customizable Messages System
1. Create `quota_messages` table migration with fields: `key`, `message`, `created_at`, `updated_at`
2. Seed default messages for each key
3. Create `QuotaMessage` model and service for managing messages
4. Add admin CRUD interface for quota messages management

### Phase 3: UI Updates
1. Update proposal index views to conditionally show create buttons
2. Add quota status indicators using customizable messages
3. Update navigation menus to hide/disable ineligible actions

### Phase 4: Testing & Validation
1. Test quota enforcement with various scenarios (draft vs submitted proposals)
2. Verify UI correctly disables buttons for ineligible users
3. Test admin CRUD for custom messages
4. Fix PHPStan static analysis errors:
   - Add return types to methods in ProposalIndex and QuotaMessageManager
   - Add value types to array parameters and return types
   - Fix undefined property access
5. Ensure existing functionality remains intact

## Technical Details

### Code Changes Required
- `app/Actions/Proposal/IdentityEligibilityAction.php`: Include DRAFT in active statuses
- `app/Services/EligibilityService.php`: Add `canCreateProposal()` method
- `app/Livewire/Abstracts/ProposalCreate.php`: Add eligibility check in `mount()`
- Database: Add `quota_messages` table for customizable messages
- Admin UI: Add CRUD interface for managing quota feedback messages
- View files: Add conditional rendering for create buttons with dynamic messages

### Database Impact
- No schema changes required
- Existing eligibility_rules JSON structure supports new global quotas

### User Experience
- Lecturers see disabled create buttons with explanatory tooltips
- Clear messaging about quota limits using admin-customizable messages with placeholders:
  - **Button Tooltip**: Admin dapat edit template seperti "Kuota terbatas: maksimal {limit} usulan sebagai ketua aktif (termasuk draft)"
  - **Access Denied Message**: Admin dapat edit template seperti "Anda telah mencapai batas maksimal {limit} usulan sebagai ketua. {custom_instruction}"
  - **Dashboard Status**: Admin dapat edit template seperti "Kuota Ketua: {current}/{limit} (termasuk {draft_count} draft)"
- Admin dapat tambah, edit, hapus narasi feedback melalui interface CRUD
- No disruption to existing draft editing workflow

## Risks & Considerations
- **Performance**: Additional eligibility checks on page load
- **UX**: Users might be confused if buttons are disabled without clear explanation
- **Edge Cases**: Handling quota changes after drafts are created
- **Backward Compatibility**: Ensure existing drafts don't break eligibility

## Success Criteria
- Draft proposals count towards quota limits
- Create proposal buttons disabled when quota exceeded
- Clear user feedback on quota status
- No breaking changes to existing functionality
