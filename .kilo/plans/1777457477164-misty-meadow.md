# Implementation Plan: Add Kaprodi Assignment and Hierarchical Research Roadmap System

## Overview
Implement hierarchical research roadmap system based on Indonesian higher education best practices (Kemdiktisaintek policies). Add Kaprodi assignment to Study Programs and enable Kaprodi to manage department-level research roadmaps that align with faculty and institutional priorities. Support checking proposal alignment with department roadmaps without disrupting existing workflows.

## Research & Community Service Workflow (Unchanged)
The core proposal workflow remains identical to support minimal disruption:

**Current Flow (Preserved):**
1. **DRAFT** → Lecturer creates/edits proposal
2. **SUBMITTED** → Lecturer submits to dekan
3. **APPROVED** (Dekan) → Initial approval
4. **WAITING_REVIEWER** → Admin LPPM assigns reviewers
5. **UNDER_REVIEW** → Reviewers evaluate
6. **REVIEWED** → All reviews complete
7. **COMPLETED/REJECTED** → Kepala LPPM final decision

**Enhancements (Additive Only):**
- Kaprodi can view proposal alignment during any stage
- Roadmap data available as reference for reviewers
- Alignment indicators shown in proposal views
- No mandatory gates or workflow interruptions

## Enhanced Workflow with Roadmap Support

**Pre-Submission Phase:**
- Kaprodi maintains program roadmap (Admin LPPM assigns kaprodi)
- Dekan maintains faculty roadmap (aligned with Renstra PT)
- Lecturers can reference roadmaps when developing proposals

**Proposal Creation (DRAFT):**
- System displays relevant program/faculty roadmaps for reference
- Lecturers encouraged to align proposals with roadmap priorities
- No mandatory validation at this stage

**Submission & Approval (SUBMITTED → APPROVED):**
- Dekan can check faculty roadmap alignment during approval
- Kaprodi can provide informal feedback on alignment
- Existing approval criteria unchanged

**Review Phase (UNDER_REVIEW):**
- Reviewers see alignment indicators with program roadmap
- Kaprodi can access proposals for alignment assessment
- Review criteria can include roadmap alignment scoring

**Post-Review (REVIEWED):**
- Final decision considers roadmap alignment feedback
- Successful proposals contribute to roadmap implementation tracking

**No Workflow Gates:** All roadmap features are informational/supporting, not blocking.

## Role Definitions & Responsibilities

**Admin LPPM:**
- Manage master data for institutions, faculties, and study programs
- Assign kaprodi to study programs
- Monitor roadmap completion across programs
- Ensure hierarchical alignment between faculty and program roadmaps
- Generate reports on roadmap implementation and proposal alignment

**Dekan (Faculty Dean):**
- Define faculty-level 5-year research roadmap aligned with university Renstra
- Determine major themes based on local excellence
- Approve and validate study program roadmaps for consistency
- Bridge LPPM policies to departmental implementation
- Monitor faculty-wide research progress

**Kaprodi (Study Program Head):**
- Develop detailed program-specific research roadmap based on CPL
- Define specific research topics, TKT progression, and success indicators
- Ensure roadmap alignment with faculty priorities and national standards
- Validate proposal alignment during review process
- Guide lecturers in developing research proposals that fit program roadmap

## Hierarchical Roadmap Context (Based on National Standards)
Following the 6-level hierarchy from National to Lecturer level:

1. **National**: PRN 2025-2029, RIRN, BIMA funding priorities
2. **University**: RIP PT, Renstra PT, IKU targets  
3. **LPPM**: Internal funding (Hibah Dosen Pemula), proposal guidelines
4. **Faculty**: 5-year Roadmap Fakultas with local excellence themes
5. **Study Program**: Detailed Roadmap Penelitian Prodi based on CPL, TKT 1-9, research tree
6. **Lecturer**: Personal 3-5 year roadmap, BIMA/SIMLITABMAS proposals

## Current System Analysis
- **Existing Structure**: Institution > Faculty > StudyProgram hierarchy
- **Master Data Management**: StudyProgramManager component in settings for CRUD operations
- **Roadmap Data**: Currently exists in Research model (per-proposal), not per-program
- **Kaprodi Role**: Not implemented in roles system yet
- **No Kaprodi Assignment**: Study programs don't have assigned department heads
- **No Hierarchical Alignment**: No connections between faculty and study program roadmaps

## Proposed Minimal Changes

### 1. Database Schema Changes
**Update study_programs table:**
- Add `kaprodi_user_id` UUID field (nullable, foreign key to users)
- Add `research_roadmap` JSON field (nullable) for hierarchical department research roadmap

**Update faculties table (if needed):**
- Add `research_roadmap` JSON field for faculty-level 5-year roadmap

**Add kaprodi role:**
- Add 'kaprodi' role to RoleSeeder if not exists
- Allow users to have both 'dosen' and 'kaprodi' roles

**Roadmap Data Structure:**
```json
{
  "period": "2025-2029",
  "priorities": [
    {
      "year": 2025,
      "themes": ["Topik 1", "Topik 2"],
      "tkt_focus": "1-3",
      "cpl_alignment": "Deskripsi keterkaitan CPL",
      "funding_sources": ["Internal", "BIMA"]
    }
  ],
  "research_tree": ["Fundamental", "Applied", "Development"],
  "success_indicators": ["Publications", "HKI", "Student involvement"]
}
```

### 4. Role-Based Access & Permissions
**Admin LPPM Permissions:**
- Full CRUD access to all master data (institutions, faculties, programs)
- Assign/unassign kaprodi to study programs
- View all roadmaps and alignment reports
- Override roadmap approvals if needed

**Dekan Permissions:**
- View and edit faculty roadmap
- Approve/reject study program roadmaps within their faculty
- View alignment reports for their faculty
- Access to faculty-level proposal alignment overview

**Kaprodi Permissions:**
- Edit roadmap for their assigned study program(s)
- View proposals from their program for alignment checking
- Provide feedback on proposal-roadmap alignment
- Access to program-specific reports and analytics

**Access Control Implementation:**
- Role-based middleware for roadmap editing
- Ownership validation (kaprodi can only edit their programs)
- Hierarchical approval workflow (faculty approves program roadmaps)

### 5. Hierarchical Roadmap Alignment Checking & Workflow

**Multi-Level Alignment Validation:**
- **Faculty-Program Alignment**: Kaprodi submits program roadmap → Dekan reviews/approves
- **Proposal-Program Alignment**: During proposal submission/review, check against program roadmap
- **TKT Progression**: Validate research TKT level matches program roadmap progression
- **CPL Connection**: Ensure research supports program learning outcomes

**Role-Specific Alignment Actions:**
- **Kaprodi**: Review proposal alignment during initial screening, provide feedback
- **Dekan**: Validate program roadmap alignment with faculty priorities
- **Admin LPPM**: Monitor overall alignment, resolve conflicts, generate compliance reports
- **Reviewers**: Consider roadmap alignment in technical evaluation

**Alignment Display & Feedback:**
- Show alignment status in proposal views with role-based indicators
- Color-coded status: Aligned (green), Partial (yellow), Requires Adjustment (red)
- Detailed feedback showing which roadmap elements match/differ
- Automated suggestions for proposal adjustment based on roadmap gaps
